How these tools were made

Several people asked how the camp's tools were designed — what the rules were, where they came from, and how to make more. This page is the answer: the principle, the constraints and the reason behind each one, the lineage, and the method baked into every tool so you can build your own.

§ A · The one principle

the test every tool has to pass

The whole design brief, in one line

Every tool should make something invisible visible. Not "teach about AI" in general — make one hidden mechanism (a probability distribution, a denoising step, drift across frames) something you can see, change, and break. A tool that explains is a slideshow; a tool that lets you turn one knob and watch the output move is an instrument. The camp builds instruments.

The companion rule is that the tools are simulations on purpose. They run on frozen examples and hand-authored distributions rather than a live model — so the mechanism stays legible, the result is the same every time you demonstrate it, and nothing depends on an account or a server. A live model would be more impressive and less teachable.

§ B · The constraints & why

each rule is a pedagogical decision in disguise
Single-file HTML
One file you can read top to bottom, save, email, or drop into a folder. No build step means a teacher can open the source and actually see how it works — the tool is its own documentation.
Vanilla JavaScript
No framework, no package install, no version rot. A tool written in plain JS still runs in five years and can be remixed by someone who only knows a little code.
No required accounts
The moment a tool needs a login, it has excluded someone — see AI access & inequality. Account-free is an equity rule first.
No live model dependency
Core teaching moments can't break because an API is down, rate-limited, or has quietly changed its behavior. Reliability in a live classroom is non-negotiable.
Frozen / precomputed examples
The same demonstration every time means you can plan a lesson around it. It also keeps the comparison honest — everyone sees the identical case.
GitHub Pages friendly & embeddable
Static files host for free and drop into a Notion page or an LMS as an iframe. Free hosting keeps the whole project inside the access rule.
A low-AI and no-AI path
Every tool works for someone who doesn't want to generate anything live — observation and critique are full pathways, not consolation prizes.

§ C · Where they came from

two bodies of work, joined

The tools grew out of a generative-AI course — ELIZA-vs-LLM comparisons, next-token prediction activities, image Default Tests, diffusion and latent-space activities, Temporal Telephone, A/B/C prompt testing, and student model-behavior investigations. The activities that survived contact with real students were the ones strong enough to adapt for adults.

The format comes from CC Fest Coding Camp: free, virtual, community-centered workshops that are beginner-friendly but intellectually serious — with recaps, asynchronous access, guest speakers, assignment tiers, and a closing showcase. Learning Machines is those two lineages fused: classroom-tested mechanisms delivered the CC Fest way. The full account is in the Project Brief.

§ D · The method inside each tool

the loop the tools are shaped to support

Every tool is built to support the same four-beat loop, the one repeated across every session:

The take-home method

Predict before you press the button · change one variable · compare the outputs · name precisely what the machine did — a default, a failure, or a pattern. The tools change; the method is the thing participants keep. Each session widens it into a full investigation: hypothesis → controlled A/B/C test → output comparison → evidence-based claim → ethical and pedagogical reflection.

§ E · Make your own

the "Explain" studio pathway

Because every tool is a single readable file with no dependencies, the most direct way to understand one is to open its source and change it — swap the frozen examples for your own, relabel a knob, or fork the structure into a new mechanism. Building an explanatory tool, poster, or concept bridge is one of the camp's final-project pathways.
